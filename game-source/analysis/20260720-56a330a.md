# 2026-07-20 上游 `56a330a` 三类问题报告

审查范围：上游 [`18a5c95b929c36a4b255ccc686e6560316cb3765`](https://github.com/RShock/zhenxun_plugin_fishing/commit/18a5c95b929c36a4b255ccc686e6560316cb3765) 到 [`56a330a258dfc4ffdd0cd1011ad8898b717cd47b`](https://github.com/RShock/zhenxun_plugin_fishing/commit/56a330a258dfc4ffdd0cd1011ad8898b717cd47b) 的[完整增量](https://github.com/RShock/zhenxun_plugin_fishing/compare/18a5c95b929c36a4b255ccc686e6560316cb3765...56a330a258dfc4ffdd0cd1011ad8898b717cd47b)，并复核开始钓鱼、捕获、鱼饵、天气与 Buff、停止结算、奖励入库和出售定价的完整链路。以下只保留三类结论。

## 游戏源码 Bug

### 1. 跨 23:00 结算会漏掉乱纪元 UTR（高）

[`is_chaotic_era_active()`](https://github.com/RShock/zhenxun_plugin_fishing/blob/56a330a258dfc4ffdd0cd1011ad8898b717cd47b/weather_service.py#L376) 虽然接收 `at_time`，查询时却固定使用当前 `_weather_date()`；[结算上下文](https://github.com/RShock/zhenxun_plugin_fishing/blob/56a330a258dfc4ffdd0cd1011ad8898b717cd47b/core/actions.py#L140) 又只用 `settle_start` 判断一次，并把结果套到整个区间。

例如结算区间为 22:50-23:10：如果新天气日从 23:00 开始，22:50 会被判定为尚未开始；如果乱纪元属于旧天气日，查询又找不到旧记录。两种情况都会漏掉实际存在的乱纪元窗口。收杆、状态结算、时光药水第一阶段和跨日期 GM 模拟均受影响。

### 2. 实际捕获与游戏内概率展示互相矛盾（高）

[实际捕获引擎](https://github.com/RShock/zhenxun_plugin_fishing/blob/56a330a258dfc4ffdd0cd1011ad8898b717cd47b/core/engine.py#L383) 只有在 `weather_lost_wind=True` 时才启用星空特殊 UTR 和 150 次保底，即“完成场景成就且当前乱纪元有效”。

但[展示概率函数](https://github.com/RShock/zhenxun_plugin_fishing/blob/56a330a258dfc4ffdd0cd1011ad8898b717cd47b/core/probability.py#L65) 使用 `weather_lost_wind or starry_utr_unlocked`；[状态页调用方](https://github.com/RShock/zhenxun_plugin_fishing/blob/56a330a258dfc4ffdd0cd1011ad8898b717cd47b/core/actions.py#L363) 完成成就后无条件传入 `starry_utr_unlocked=True`。因此太阳风、流星雨和恒纪元期间实际 UTR 为 0，页面仍显示非零 UTR。

### 3. 达到 1200 分不会实际发放 S2 入场券（高）

[`s2_ticket_claimed`](https://github.com/RShock/zhenxun_plugin_fishing/blob/56a330a258dfc4ffdd0cd1011ad8898b717cd47b/models/user.py#L209) 只有字段定义和读取，没有写为 `True` 的业务路径。[收杆逻辑](https://github.com/RShock/zhenxun_plugin_fishing/blob/56a330a258dfc4ffdd0cd1011ad8898b717cd47b/core/actions.py#L689) 仅计算 `reached` 并展示进度，即使累计分数达到 1200 也不会发券，与当前玩法文档“达到后获得”不符。

### 4. 收杆事务外的“纯计算”会写入当前鱼饵（中）

[自动换饵](https://github.com/RShock/zhenxun_plugin_fishing/blob/56a330a258dfc4ffdd0cd1011ad8898b717cd47b/core/engine.py#L946) 在模拟期间直接保存 `user.bait_id`；而[收杆函数](https://github.com/RShock/zhenxun_plugin_fishing/blob/56a330a258dfc4ffdd0cd1011ad8898b717cd47b/core/actions.py#L879) 明确声明事务外模拟只读。若后续模拟或事务失败，鱼获和鱼饵库存会回滚或不提交，但当前鱼饵可能已经改变。

状态查询路径还会[先保存结算状态、再扣鱼饵](https://github.com/RShock/zhenxun_plugin_fishing/blob/56a330a258dfc4ffdd0cd1011ad8898b717cd47b/core/actions.py#L293)。两次写入之间失败时，鱼获已经记入待领取状态，鱼饵却未扣除。

### 5. 150 次保底奖励正确，但触发提示少一次（低）

[实际保底](https://github.com/RShock/zhenxun_plugin_fishing/blob/56a330a258dfc4ffdd0cd1011ad8898b717cd47b/core/engine.py#L400) 使用 `pity + 1 >= 150`，而[提示预判](https://github.com/RShock/zhenxun_plugin_fishing/blob/56a330a258dfc4ffdd0cd1011ad8898b717cd47b/core/engine.py#L1180) 使用 `pity >= 150`。计数为 149 时第 150 次会正确获得奖励，但不会显示保底触发提示；展示木框提示同样存在此偏差。

### 6. 当前游戏帮助仍描述旧规则（中）

目标提交已经把星空 UTR 改为乱纪元期间生效并取消星辰木框持有上限，但 [`doc.md`](https://github.com/RShock/zhenxun_plugin_fishing/blob/56a330a258dfc4ffdd0cd1011ad8898b717cd47b/doc.md#L602) 仍写“乱纪元无效果”“成就后 UTR 常驻”，并在[木框章节](https://github.com/RShock/zhenxun_plugin_fishing/blob/56a330a258dfc4ffdd0cd1011ad8898b717cd47b/doc.md#L712) 引用已删除的 `STAR_FRAMES_MAX`。网页帮助也保留相同旧口径，会直接误导玩家。

## 利润网 Bug

### 1. 星空特殊 UTR 被错误地按成就完成后全天计入（高）

利润网的 [`isSpecialUtrEnabled()`](../../catch-outcome.js) 对星空图只检查场景成就，不检查天气类型和有效期；现有[回归测试](../../tests/catch-outcome.test.js) 还明确断言太阳风期间特殊 UTR 生效，[PRD](../../PRD.MD) 也写成“常驻特殊掉落”。

按照目标源码的实际捕获链，正确条件是“当前图场景成就完成 + 当前乱纪元有效”。太阳风、流星雨、恒纪元以及尚未开始或已经结束的乱纪元均不产生特殊 UTR，150 次 UTR 保底也冻结。源码每天恰有 5/10 张星空图为乱纪元，因此固定地图按长期地图日统计时，利润网当前会把这部分 UTR 随机收益和保底推进约高估一倍。

### 2. 缺失或非法星空天气被强制标记为有效乱纪元（中）

游戏源码在没有天气记录时返回“显示为乱纪元，但 `is_active=False`”；利润网的 [`getWeatherForMap()`](../../app.js) 将星空图的缺失或非法天气归一为乱纪元时，同时强制写入 `is_active: true` 并清空起止时间。

这会把“仅用于占位显示的乱纪元”误认为正在生效。当前它部分被“星空 UTR 全天启用”的第一个 Bug 掩盖；若只修正 UTR 天气门控而不同时修正这里，缺失天气数据时仍会错误计入 UTR。

## 源码与利润网有差异但无法确定是否为 Bug

### 1. 星辰木框取消上限和自动奇迹消费

源码现在允许无限持有星辰木框，并在收杆时[自动连续消费可组成奇迹的流星鱼](https://github.com/RShock/zhenxun_plugin_fishing/blob/56a330a258dfc4ffdd0cd1011ad8898b717cd47b/models/user_mutations.py#L508)。星空木框仍最多升级 10 个，因此满级后继续消耗流星鱼、积累星辰木框是否是设计意图无法从代码确定。

利润网没有给流星鱼、奇迹或星辰木框统一折算鱼币。是否应计算这部分机会成本，取决于这些资源是否要纳入净利润定义。

### 2. 星空评分源码指纹变化，但评分规则未变化

目标提交只从 `core/starry_system.py` 删除了与评分无关的星辰木框上限常量；评分函数、编号域和奖池均未变化，利润网现有 PMF 数值仍正确。但整文件 SHA-256 已变化，现有生成器的源码漂移保护会失败。

这是需要人工确认后更新的来源指纹，不是已确认的收益算法 Bug；是否接受新指纹应与是否更新利润网代码分开决定。
