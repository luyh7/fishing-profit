# 20260719_134826 新版源码与利润网复核

> 历史快照说明：本报告对应压缩包快照，不对应当前上游子模块。报告使用的源码固定保存在利润网提交 [`267d99c`](https://github.com/luyh7/fishing-profit/tree/267d99c/game-source/current) 中。

## 结论

新版游戏源码已经修复上一版的高鱼竿实际结算错误：完整 16 槽概率会在抽样前归并到地图稀有度上限，扩展槽不再回退为 N。S1 猫猫乐园的普通表上限也由 UTR 改为 UR，不再自然产出 UTR。

当前利润网仍按 20260717 版实际引擎建模，因此这两项现在反而成为新的高影响不一致：

- 普通 UR 图在 `d=11` 时，利润网仍给出 N 7.04%；新版源码为 N 0%、UR 89.15%。
- 普通 UR 图在 `d>=17` 时，利润网仍为 N 100%；新版源码为 UR 100%。
- S1 在 `d=9` 时，利润网仍给普通表 UTR 11.37%；新版源码普通表 UTR 为 0%，该质量并入 UR。
- 利润网仍按旧版“原始槽先流星、再运行时映射”处理；新版实际结算已改为“先按地图上限归并、再流星”。

旧报告中用户点名的其他规则大体仍与实际结算一致，包括 11-20 图成就后常驻特殊 UTR、幸运取优、幸运与保底顺序、多多与逗猫棒数量、猫猫乐园鱼价取整、猫天气和礼物规则。需要注意，概率封顶发生在幸运双抽之前，利润网保留“封顶前稀有度”作为取优优先级的做法也已经过时。

新版还取消了展示木框和猫猫框出售。展示木框按 0 计价现在与“无直接出售收入”一致；猫猫框 200 不再是源码售价，只能视为用户指定估值。

源码自身仍有问题：实际流星路径在 `d>=12` 会形成 `N=-2% / UR=102%` 的内部数组；状态页流星展示仍走旧的 raw-first 路径，并可能在普通 UR 图展示不可能由普通表产出的 UTR。本文以实际结算路径为利润网对齐口径，同时单独记录这些源码展示、文档和实际执行的不一致。

## 新版实际结算调用链

一次捕获的关键顺序如下：

1. `FishingBuffCalculator.get_effects_at_time` 计算有效鱼竿、鱼饵、天气和药水；多多令有效鱼竿减 1，见 [`models/buff.py:463`](https://github.com/luyh7/fishing-profit/blob/267d99c/game-source/current/models/buff.py#L463) 和 [`models/buff.py:511`](https://github.com/luyh7/fishing-profit/blob/267d99c/game-source/current/models/buff.py#L511)。
2. 11-20 图未完成 `collect_scene_<id>` 时把运行时上限压到 UR；完成后保留配置中的 UTR，见 [`engine.py:1009`](https://github.com/luyh7/fishing-profit/blob/267d99c/game-source/current/core/engine.py#L1009)。
3. S1 加载材料率、逗猫棒、城堡、鱼饵节省和过山车天气增幅，见 [`engine.py:1016`](https://github.com/luyh7/fishing-profit/blob/267d99c/game-source/current/core/engine.py#L1016)。
4. 水晶猫城堡每次捕获只判定一次鱼竿 `+1`；幸运药水两个随机结果共用该等级，见 [`engine.py:525`](https://github.com/luyh7/fishing-profit/blob/267d99c/game-source/current/core/engine.py#L525)。
5. 先计算多多/逗猫棒数量，再依次检查展示木框保底、特殊 UTR 保底；触发保底时跳过幸运双抽，见 [`engine.py:380`](https://github.com/luyh7/fishing-profit/blob/267d99c/game-source/current/core/engine.py#L380) 和 [`engine.py:400`](https://github.com/luyh7/fishing-profit/blob/267d99c/game-source/current/core/engine.py#L400)。
6. 未触发保底时，每个随机结果依次判定 S1 材料、非星空图 0.7% 展示木框、特殊 UTR、普通稀有度表，见 [`engine.py:176`](https://github.com/luyh7/fishing-profit/blob/267d99c/game-source/current/core/engine.py#L176)。
7. 普通稀有度表先归并到地图上限，再应用流星，最后抽样，见 [`engine.py:214`](https://github.com/luyh7/fishing-profit/blob/267d99c/game-source/current/core/engine.py#L214)。
8. 幸运从两个完整随机结果中取优，随后只按最终结果更新一次保底，见 [`engine.py:439`](https://github.com/luyh7/fishing-profit/blob/267d99c/game-source/current/core/engine.py#L439) 和 [`engine.py:481`](https://github.com/luyh7/fishing-profit/blob/267d99c/game-source/current/core/engine.py#L481)。

## 利润网新出现的不一致

### 1. 高渔力仍把扩展槽映射为 N

利润网在 [`catch-outcome.js:104`](/home/luyh7/game/fishing-profit/catch-outcome.js:104) 遍历原始槽，并在 [`catch-outcome.js:109`](/home/luyh7/game/fishing-profit/catch-outcome.js:109) 将所有 `rawIndex>=6` 映射为索引 0，即 N；之后的地图封顶无法把 N 恢复为 UR。

确定性差分如下。表内只列普通鱼稀有度，不含材料、展示木框和特殊 UTR：

| 场景 | 新版实际结算 | 当前利润网 |
| --- | --- | --- |
| 图1 Lv.11，`d=11` | SSR 10.85% / UR 89.15% | N 7.04% / SSR 10.85% / UR 82.11% |
| 图1 Lv.12，`d=12` | UR 100% | N 18.08% / UR 81.92% |
| 图1 Lv.17，`d=17` | UR 100% | N 100% |
| 图1 Lv.19，`d=19` | UR 100% | N 100% |

图1 `d>=17` 时，单条普通鱼稀有度期望倍率因此从源码的 16 被利润网算成 1，低估 93.75%。这正是新版已经修掉、但当前页面仍会显示的“高杆收益下降”。

### 2. S1 仍配置为 UTR 上限

利润网 [`config.js:491`](/home/luyh7/game/fishing-profit/config.js:491) 仍写 `maxRarity: "UTR"`，与新版 `locations.json` 的 UR 冲突。

S1 鱼竿 Lv.15、difficulty 6、`d=9`，普通天气下：

| 口径 | SR | SSR | UR | UTR |
| --- | ---: | ---: | ---: | ---: |
| 新版实际结算 | 6.77% | 37.74% | 55.49% | 0% |
| 当前利润网 | 6.77% | 37.74% | 44.12% | 11.37% |

普通鱼稀有度期望倍率分别为 12.1684 和 13.9876，利润网高估约 14.95%。真正的 S1 UTR 还应另走“成就解锁后的当前迷途风特殊概率 + 150 保底”。

### 3. 流星仍使用旧版顺序

新版实际路径是：

`原始16槽 -> 按地图上限归并 -> 流星搬移 -> 抽样`

利润网则在 [`catch-outcome.js:93`](/home/luyh7/game/fishing-profit/catch-outcome.js:93) 先对原始数组应用流星，再从 [`catch-outcome.js:104`](/home/luyh7/game/fishing-profit/catch-outcome.js:104) 开始执行旧版运行时映射。

图1 Lv.9、`d=9`、流星 +2%：

| 口径 | SR | SSR | UR | UTR/N |
| --- | ---: | ---: | ---: | ---: |
| 新版实际结算 | 6.77% | 35.74% | 57.49% | 0% |
| 当前利润网 | 6.77% | 37.74% | 55.49% | 0% |
| 新版源码状态页 | 6.77% | 37.74% | 42.12% | UTR 13.37% |

利润网的流星加成在该边界被同档封顶抵消；新版实际会把 2 个百分点从 SSR 移到 UR。

### 4. 幸运仍保留封顶前稀有度优先级

新版先合并概率槽再抽样，因此由原 UR、UTR 或更高扩展槽并入的质量在抽样时都只是同一个 UR 槽，幸运取优不再知道它们来自哪个原始槽。

利润网仍在 [`catch-outcome.js:112`](/home/luyh7/game/fishing-profit/catch-outcome.js:112) 保存 `selectionRarity`，并在 [`catch-outcome.js:306`](/home/luyh7/game/fishing-profit/catch-outcome.js:306) 用它决定幸运优先级。于是同为最终 UR 的结果仍可能因“原始 UTR”而被优先选择。没有收集数据时通常不改变售价，但在同稀有度未收集鱼取优时会改变鱼种选择和期望价值。

### 5. 现有测试固化了旧版规则

利润网 23 个测试均通过，但以下测试现在断言的是旧版行为：

- S1 普通表保留 UTR、高阶槽回退 N：[`catch-outcome.test.js:171`](/home/luyh7/game/fishing-profit/tests/catch-outcome.test.js:171)。
- 普通图 `d=12` 仍有 N 18.08%：[`catch-outcome.test.js:199`](/home/luyh7/game/fishing-profit/tests/catch-outcome.test.js:199)。
- 流星先调整 raw 槽再映射：[`catch-outcome.test.js:222`](/home/luyh7/game/fishing-profit/tests/catch-outcome.test.js:222)。
- `d=19` 明确断言 N 100%：[`catch-outcome.test.js:258`](/home/luyh7/game/fishing-profit/tests/catch-outcome.test.js:258)。

因此测试通过只证明当前实现内部自洽，不能证明已对齐新版。

## 仍保留的差异

旧报告列出的 13 项未点名差异在新版中都仍存在；本轮没有修改利润网。

### 1. 24 小时尾段仍向下取整

游戏对不足一个完整间隔的尾段按 `remaining / interval` 再做一次 Bernoulli 捕获：[`engine.py:801`](https://github.com/luyh7/fishing-profit/blob/267d99c/game-source/current/core/engine.py#L801)。利润网仍用 `floor(24 / interval)`：[`app.js:2672`](/home/luyh7/game/fishing-profit/app.js:2672)。因此最多漏掉不足 1 次鱼获及对应成本。

### 2. 地图准入道具未建模

游戏中 S1 需要门票但不再检查杆级；11-20 图需要星空艇并检查杆级：[`actions.py:171`](https://github.com/luyh7/fishing-profit/blob/267d99c/game-source/current/core/actions.py#L171)。利润网只按原始鱼竿与难度过滤：[`app.js:2714`](/home/luyh7/game/fishing-profit/app.js:2714)。

### 3. 库存、换饵与逐时状态未模拟

游戏按结算时间点重新计算 buff，并在鱼饵耗尽后切换到库存中价格最高的鱼饵，最终进入无饵模式：[`engine.py:950`](https://github.com/luyh7/fishing-profit/blob/267d99c/game-source/current/core/engine.py#L950)、[`engine.py:981`](https://github.com/luyh7/fishing-profit/blob/267d99c/game-source/current/core/engine.py#L981)。利润网把当前状态外推 24 小时并假设候选鱼饵无限。

### 4. 其他药水和 buff 未计入

游戏仍支持许愿、加速、双倍捕获、闪光等效果；利润网配置只提供无药水、多多和幸运：[`config.js:47`](/home/luyh7/game/fishing-profit/config.js:47)。这些效果会改变速度、数量或额外奖励。

### 5. 星空鱼和星空天气额外奖励未计入

新版星空鱼掉率为：

`5% + max(0, effectiveRodLevel - 10) * 0.5% + (solarWind ? 2.5% : 0)`

各项是绝对百分点相加，见 [`starry_system.py:543`](https://github.com/luyh7/fishing-profit/blob/267d99c/game-source/current/core/starry_system.py#L543)；实际捕获会传入有效鱼竿并生成奖励：[`engine.py:600`](https://github.com/luyh7/fishing-profit/blob/267d99c/game-source/current/core/engine.py#L600)。幸运和流星雨会增加候选编号，多多会复制同号星空鱼，新版还增加对子/葫芦评分和碎片奖池。利润网仍完全没有折算这些收益。

### 6. 首条 UTR 消耗和一次性成就收益未计入

第一条未收集 UTR 会被自动消耗以解锁图鉴，不进入可出售背包；其余数量才入包：[`result.py:60`](https://github.com/luyh7/fishing-profit/blob/267d99c/game-source/current/core/result.py#L60)。利润网把所有 UTR 都按可售鱼计入稳态收益，也不计一次性成就金币。

### 7. 猫礼物的同稀有度选鱼策略不同

源码优先鱼池中的第一条未收集鱼，全部收集后才选最贵鱼：[`cat.py:67`](https://github.com/luyh7/fishing-profit/blob/267d99c/game-source/current/core/cat.py#L67)。利润网未完成场景成就时仍使用同稀有度随机鱼平均值，完成后才使用最贵鱼：[`app.js:2590`](/home/luyh7/game/fishing-profit/app.js:2590)。

### 8. 猫猫框 15 次保底未建模

源码在每次猫吃鱼时推进猫框计数，达到 15 次直接给框：[`cat.py:25`](https://github.com/luyh7/fishing-profit/blob/267d99c/game-source/current/core/cat.py#L25)。利润网只计算 15% 随机猫框及幸运额外判定：[`catch-outcome.js:815`](/home/luyh7/game/fishing-profit/catch-outcome.js:815)。

### 9. 手动猫猫乐园建筑可形成非法组合

源码用传奇猫雕像等级门控其他建筑升级：[`cat_park.py:257`](https://github.com/luyh7/fishing-profit/blob/267d99c/game-source/current/cat_park.py#L257)。利润网页面只在升级雕像本身时检查其他八栋建筑，并允许手动降级：[`app.js:4729`](/home/luyh7/game/fishing-profit/app.js:4729)、[`app.js:5644`](/home/luyh7/game/fishing-profit/app.js:5644)。自动同步的真实玩家状态不受影响。

### 10. 页面使用长期保底稳态，不读取当前计数

状态 API 的玩家负载仍未提供 `frame_pity_counter`、`utr_pity_counter` 或 `cat_frame_pity_counter`：[`status_api.py:502`](https://github.com/luyh7/fishing-profit/blob/267d99c/game-source/current/status_api.py#L502)。利润网继续用长期稳态 Markov 分布：[`catch-outcome.js:597`](/home/luyh7/game/fishing-profit/catch-outcome.js:597)。临近保底玩家的短期结果会不同。

### 11. 幸运与水晶猫城堡的相关性不同

游戏每次捕获只判一次城堡，然后幸运两次随机结果共用鱼竿等级：[`engine.py:525`](https://github.com/luyh7/fishing-profit/blob/267d99c/game-source/current/core/engine.py#L525)。利润网先混合“基础等级/等级+1”两档分布，再做独立幸运取优：[`app.js:2428`](/home/luyh7/game/fishing-profit/app.js:2428)。

### 12. 手动模式没有星空成就输入

星空特殊 UTR 只从自动玩家快照的 `achievements` 判断：[`app.js:831`](/home/luyh7/game/fishing-profit/app.js:831)、[`app.js:3566`](/home/luyh7/game/fishing-profit/app.js:3566)。未匹配玩家无法手动声明各星空图已完成成就，当前会按未解锁处理。

### 13. 过山车天气增幅未作用于迷途风 UTR

游戏会把过山车增幅写入 `weather_lost_wind_multiplier`：[`engine.py:1036`](https://github.com/luyh7/fishing-profit/blob/267d99c/game-source/current/core/engine.py#L1036)。利润网特殊 UTR 只混合城堡的鱼竿等级，没有乘天气增幅：[`app.js:843`](/home/luyh7/game/fishing-profit/app.js:843)。

### 14. 手动迷途风可绕过普通图成就门槛

游戏会在加载 buff 时过滤未完成场景成就玩家的迷途风：[`models/buff.py:241`](https://github.com/luyh7/fishing-profit/blob/267d99c/game-source/current/models/buff.py#L241)。利润网手动天气覆盖在成就门控之前直接返回：[`app.js:680`](/home/luyh7/game/fishing-profit/app.js:680)，而非星空图的特殊 UTR 开关不检查 `hasAchievement`：[`catch-outcome.js:849`](/home/luyh7/game/fishing-profit/catch-outcome.js:849)。因此手动选择迷途风时，1-10/S1 即使未完成 `collect_scene_<id>` 也会计算 UTR。

## 用户指定和有意差异

- 展示木框价值固定为 0。新版已删除其出售入口，直接现金价值同为 0；框仍可用于速度 buff 和升级展示栏位，这部分非现金用途未估值。源码说明见 [`doc.md:191`](https://github.com/luyh7/fishing-profit/blob/267d99c/game-source/current/doc.md#L191)。
- 猫猫框价值固定为 200。新版同样不可出售，源码没有固定现金等价；利润网的 200 是用户指定估值，不是新版源码售价。
- 利润网页面定位仍是“当前配置保持不变时的 24 小时稳态比较”，不是背包、天气、药水和保底计数的逐时复刻。由此产生的已知差异列在上一节。

## 源码自身仍有问题

### 1. 流星在高差值下形成负概率

实际路径先把 `d>=12` 归并成 UR 100%，再调用 `apply_meteor_effect`。该函数无条件给最高非零槽 +2%、给相邻槽 -2%；相邻槽原本为 0 时会把负数转移到 N 后再清零相邻槽：[`constants.py:270`](https://github.com/luyh7/fishing-profit/blob/267d99c/game-source/current/constants.py#L270)。

| d | 流星调整后的内部数组 | 当前实际抽样 |
| --- | --- | --- |
| 11 | SSR 8.85% / UR 91.15% | 同左 |
| 12 | N -2% / UR 102% | UR 100% |
| 17 | N -2% / UR 102% | UR 100% |

数组总和仍为 100%，但违反概率非负性。当前累计抽样仍会全部落到 UR，不会重新出现 N 上升；任何直接按数组计算期望的调用方则可能得到无效数据。新版高槽测试尚未覆盖“归并后再流星”的边界。

### 2. 流星展示仍使用旧的 raw-first 路径

`calculate_display_probabilities` 先正确计算带上限、带多多的分布，但有流星时会把结果整个替换为原始槽计算：[`probability.py:31`](https://github.com/luyh7/fishing-profit/blob/267d99c/game-source/current/core/probability.py#L31)。`get_meteor_adjusted_probabilities` 在 raw 16 槽上先流星，再把 index 5 及以上汇总为 UTR，且普通 UR 图不重新封顶：[`constants.py:291`](https://github.com/luyh7/fishing-profit/blob/267d99c/game-source/current/constants.py#L291)。

图1 Lv.11、`d=11`、流星 +2%：

- 实际结算：SSR 8.85% / UR 91.15%。
- 源码状态页：SSR 10.85% / UR 44.26% / UTR 44.89%。

状态页会展示普通概率表不可能产出的 UTR。该替换还丢掉 `duoduo_count` 数量倍率：无流星时多多展示总和为 200%，有流星后回到 100%。

### 3. 过山车 UTR 增幅没有进入源码概率展示

实际结算会放大迷途风特殊 UTR：[`engine.py:1036`](https://github.com/luyh7/fishing-profit/blob/267d99c/game-source/current/core/engine.py#L1036)。展示函数只有 `weather_lost_wind: bool`，没有 multiplier 参数：[`probability.py:12`](https://github.com/luyh7/fishing-profit/blob/267d99c/game-source/current/core/probability.py#L12)。例如 S1 Lv.15、过山车 Lv.3，实际条件概率为 1.1%，展示仍为 1.0%。

### 4. 保底触发提示有一位偏差

真实保底条件为 `(pity + 1) >= 150`：[`engine.py:403`](https://github.com/luyh7/fishing-profit/blob/267d99c/game-source/current/core/engine.py#L403)、[`engine.py:411`](https://github.com/luyh7/fishing-profit/blob/267d99c/game-source/current/core/engine.py#L411)。主循环调用前却用 `pity >= 150` 标记本次是否保底：[`engine.py:1169`](https://github.com/luyh7/fishing-profit/blob/267d99c/game-source/current/core/engine.py#L1169)。计数为 149 时奖励会正确触发，但不会追加“保底触发”消息；奖励结算本身不受影响。

### 5. 文档存在陈旧或互相矛盾的说明

- UTR 主规则已正确写 S1 普通表上限为 UR，但猫猫乐园章节仍写 `max_rarity=UTR`：[`doc.md:112`](https://github.com/luyh7/fishing-profit/blob/267d99c/game-source/current/doc.md#L112)、[`doc.md:543`](https://github.com/luyh7/fishing-profit/blob/267d99c/game-source/current/doc.md#L543)。实际配置以 `locations.json` 的 UR 为准。
- 目录仍写“卖出展示木框/猫猫框”，正文已改为不可出售：[`doc.md:9`](https://github.com/luyh7/fishing-profit/blob/267d99c/game-source/current/doc.md#L9)、[`doc.md:191`](https://github.com/luyh7/fishing-profit/blob/267d99c/game-source/current/doc.md#L191)。
