# 一、核心系统

## 鱼添加统一接口

所有向用户添加鱼的场景均通过 `add_fish_to_user()` 统一处理，确保 UTR 自动消耗和成就检查逻辑一致。

### 送鱼

`赠送/送鱼 鱼ID @某人` 必须使用平台真实 @：手机端长按头像，电脑端输入 @ 后选择指定人；直接打字 @名字 不会被识别。

### 调用场景

| 场景 | 入口 | check_achievements |
|------|------|--------------------|
| 钓鱼收杆 | `save_fish_to_backpack` → `add_fish_to_user` | True |
| 时光药水 | `use_time_potion_settle` 写入钓鱼状态，收杆时 `save_fish_to_backpack` → `add_fish_to_user` | True |
| 送鱼 | `gift_fish` → `add_fish_to_user` | True |
| GM添加 | `gm_add_item` → `add_fish_to_user` | True |
| GM发鱼 | `gm_give_fish` → `add_fish_to_user` | True |
| 猫礼物鱼 | `stop_fishing` → `add_fish_to_user` | False（收杆时统一检查） |

### UTR 自动消耗规则

玩家首次获得某条鱼的 UTR 稀有度时，自动消耗 1 条解锁图鉴，不进入背包。剩余条数正常放入背包。

### 自动展示与成就检查

`add_fish_to_user` 在鱼实际进入背包后统一尝试自动放入展示栏；UTR 首获被消耗解锁图鉴时不展示，若有剩余条数则剩余鱼可展示。

`add_fish_to_user` 在添加完所有鱼后统一调用 `check_all_achievements` 检查全部场景成就。`check_achievements=False` 时跳过（用于猫礼物鱼等后续会统一检查的场景）。

### 旧数据修复

历史通过 GM 添加但未触发成就的数据，可使用 `gm补成就` 命令手动补发。

## 稀有度与等级差值

- 常规鱼稀有度由 **差值 `d = rod_level - location.difficulty`** 索引 `RARITY_DISTRIBUTION[d]`（状态页「鱼力」同此值）。
- 1 图 difficulty=0、2 图=1 …；S1 difficulty=6。
- **UR 截断**：`max_rarity=UR` 时，概率表 index≥5（UTR 及扩展位）质量全部并入 UR，不会通过概率表自然掉 UTR。

## UTR 概率机制

- 普通场景 `max_rarity=UR` 时，稀有度表即使抽到 UTR 位也会被封顶为 UR，不能依赖原概率表自然产出 UTR。
- **1-10 图与 S1 双重要求**：必须先集齐该图全部 UR（完成 `collect_scene_{id}` 解锁迷途风），**并且**在迷途风天气下钓鱼；只集齐 UR 或只遇到迷途风都不够。
- 普通场景 UTR 只走迷途风特殊判定：基础概率 0.2%，玩家鱼竿等级每领先场景等级 1 级，额外 +0.1%（场景等级=内部 difficulty+1，如图1为1级；等价于内部等级差 d=6 时为 0.7%）。
- 该概率需同时用于实际结算和概率展示，避免显示与掉落不一致。
- 迷途风 UTR 保底计数只要累计过（`utr_pity > 0`）就应在收杆/状态提示中显示，不要求当前仍处于迷途风天气；出 UTR 后计数归零才不显示。

## 图片资源命名

- 鱼图与场景图只使用正式 png 文件名
- 场景图文件名：`{id}-{名称}-{布局后缀}.png`（`id` 可为数字或 `s1`）
- **锚点基准（脚部）**：场景坐标表示**人物脚底**应附着的位置；角色贴图底部是钓竿末端，通过皮肤文件名 `id_{y_offset}.png` 的 `y_offset`（CSS `bottom` 像素修正）把脚对齐到锚点。2026-07 起统一：旧场景锚点 y 相对历史钓竿基准 **−13**，皮肤 `y_offset` 相对历史再 **−21**（先 −13 对齐、再累计 −8 微调下移）；旧场景锚点 y：先相对历史 −13，后因方向误调 −4，再反向 +8，再 **回拉 −2**（净相对历史 **−11**）；**仅 13 图**布局 y 始终不平移（脚部点轨）。
- **旧格式（单轨高度）**：最后一段为人物站立高度（y%）。单数字=统一高度；多个数字用 `_` 分隔，按图片横向 0%~100% 均匀作为控制点，人物根据横向位置对高度线性插值。解析失败默认 `50`
  - 例：`12-月环港-74.png`、`11-牛奶河-62_52.png`（场景 y 减小 → 角色上移）
- **新格式（多轨道轨迹）**：最后一段以 `T@` 开头（或历史 `T{数字}@`）时启用。不设人数容量，支持 1~无限人
  - 语法：`T@{x},{y}_{x},{y}+T@{x},{y}_...`
  - `@` 后为控制点序列，点与点用 `_` 连接，每个点为 `x,y`（相对场景宽高的百分比 0~100）
  - 轨道之间用 `+` 连接
  - **智能分配**：按各轨折线弧长计算长度；每次把下一个人放到「当前人数/长度」最小的轨道（密度均衡）；平局优先靠前轨道
  - **轨内等距**：1 人居中；多人沿弧长从起点到终点等间距（含端点），线性插值得到 (x,y)
  - 渲染时 `scene_bottom = 100 - y`；y 更大（更靠下）的角色 z-index 更高，用于前后遮挡
  - 姓名自动错开：`templates/fishing_scene.html` 在 `document.fonts.ready` 后按碰撞检测上移/水平校正 `.pname`，与布局格式无关
  - 例：`13-彗尾瀑-T@33.6,82.8_57.8,89.3+T@56,78_60.8,79+...`（脚部锚点，不参与 −13 平移）
- **兼容**：
  1. 无 `T…@` → 旧高度格式 `83` / `75_65`
  2. 历史 `T4@...`（带容量数字）仍可解析，容量数字被忽略
- **在线轨道编辑器**：`http://<host>:4159/_lab/track-editor/` 或 `.../index.html`（手机优先布局，大按钮/大字号）。可选地图、点选/拖拽编辑多条轨道，导出准确文件名到剪贴板。列表优先 `GET /api/scenes`，未重启时回退静态 `scenes.json`
- **文件名坐标**：接近整数（误差 ≤0.1）四舍五入为整数，其余保留 1 位小数，避免路径过长导致 Windows/`file://` 裂图
- 实现见 `render/fishing_scene.py`：`_parse_scene_layout` / `_allocate_track_slots` / `_place_on_tracks` / `_place_on_heights`
- 超管调试指令 `测试场景渲染 ID` 可随机渲染 8 个角色到指定场景图上，ID 支持数字地图与 S1，用于检查多轨道分布、遮挡与姓名错开

## 网页端图片缩放

- 钓鱼场景和建设猫猫乐园涉及场景与人物/设施的相对位置，必须严格等比例缩放：按结果区可用宽度同时缩放宽度和高度，保持比例正确
- 背包、天气、商店、图鉴等普通页面按结果区可用宽度重排铺满，可改变页面整体宽高比例，但不得使用非等比拉伸/压缩文字或图片
- 各页面宽度尽量占满结果区且不超过；高度较高时通过结果区纵向滚动浏览

## 网页端体验

- 主界面采用清爽现代风格：玻璃质感顶栏、卡片化三栏布局、统一圆角/阴影/状态色
- 桌面端保持背包 / 场景结果 / 角色状态三栏；手机端左右栏改为抽屉，主操作区优先显示
- 网页端场景可见性与 PC 端一致：未解锁猫猫乐园不显示 S1，未建设星空艇不显示 11-20
- 场景卡片展示天气、人数、打窝和展示框加成；已锁定/钓鱼中状态弱化并禁用点击
- 背包与状态信息使用资产卡片、稀有度标签和空状态提示，减少密集文本带来的阅读负担
- 4159 静态页中 `index.html` 为操作面板，`help.html` 为并列帮助页；帮助页顶部通过 `/index.html` 跳转到网页版，需包含玩家指令、活动/地图解锁条件、GM 指令说明，并提供目录锚点跳转到详细说明
- 网页端密钥只存入数据库表 `fishing_web_key`，注册、查看、删除和网页端认证均从数据库读取，不依赖 JSON 文件或代码目录

## 鱼饵管理

### 设定鱼饵

- 指令：`设定鱼饵 鱼饵ID/名称`
- 设定优先使用的鱼饵，钓鱼时优先消耗该鱼饵，用完后自动切换为最佳鱼饵
- `设定鱼饵 取消/0/自动/清除` 可清除设定，恢复自动选择最佳鱼饵
- 设定后即使当前未持有，后续购买仍会优先使用
- 存储：`FishingUser.preferred_bait_id`（默认"0"=自动选择）

### 鱼饵消耗与时间匹配

- 每个鱼循环严格按以下顺序执行：**确认鱼饵**（用完自动切换或进入无饵模式）→ **计算时间间隔**（用当前鱼饵速度）→ **钓鱼** → **钓到鱼才扣鱼饵**
- 鱼饵消耗统一后置到钓鱼后：记录钓鱼前的鱼数量，钓鱼后比较新增数量，有新增才扣鱼饵。鱼饵消耗与鱼获严格 1:1 匹配
- 剩余时间不足一个完整间隔时：按 `remaining_time / fishing_interval` 概率判定是否钓到鱼，钓到才扣鱼饵
- 结算时直接使用 `user.bait_id`，不重新选择鱼饵；开始钓鱼时已通过 `select_bait_with_preference` 设定 `bait_id`
- 钓鱼期间玩家通过"设定鱼饵"修改的是 `preferred_bait_id`，不影响正在使用的 `bait_id`，下次开始钓鱼时才生效
- 鱼饵用完时由循环内部自动切换到下一个最好的，并更新 `user.bait_id`，后续循环按新鱼饵速度计算——这是允许的机制
- 剩余时间不足一个完整间隔时：先按 `remaining_time / fishing_interval` 概率判定是否钓到 → 只有判定通过才消耗鱼饵 → 再钓鱼。鱼饵消耗与鱼获严格 1:1 匹配
- 幸运药水的两次结算发生在钓鱼阶段内部，不额外消耗鱼饵
- 鱼饵消耗统一走 `_consume_one_bait` 函数，处理暴风天气节省、猫猫乐园鱼饵节省等概率

### 卖出鱼饵

- 指令：`卖出鱼饵 鱼饵ID/名称`
- 卖出所有指定的鱼饵，按购买价原价回收
- 若卖出的鱼饵恰好是当前优先鱼饵或使用中鱼饵，自动清除设定并切换

### 展示木框 / 猫猫框不可出售

- 展示木框与猫猫框均**不可出售**（相关指令与卖出逻辑已移除）
- 展示木框：全局速度 buff、升级展示栏位
- 猫猫框：强化展示栏位；在 11-20 星空图可通过 `钓鱼使用 猫猫框 [数量]` 打窝

## 速度加成显示

钓鱼状态页面的速度加成拆分为：鱼钩 / 鱼饵 / Buff（打窝+加速药水+展示框）/ 额外乘数（周末等）/ 天气乘数（雨天等）。

- `check_fishing_status`（actions.py）：完整拆分显示
- `use_time_potion_settle`（potion.py）：与 check_fishing_status 保持一致的拆分逻辑

鱼饵速度加成以 `shop.json` 为准：蚯蚓+20%、虾米+40%、拟饵+60%、黄金+80%、魔法+100%、传说+120%。

- 速度公式：`(1 + 鱼钩%) × (1 + (鱼饵% + 打窝% + 展示木框% + 星空艇%) / 100) × 天气乘数 × 猫猫乐园乘数`

- 展示木框与猫猫框均**不可出售**

### 猫猫乐园 buff 作用范围

## 时光药水

- 闪光药水（`钓鱼使用 闪光药水 [数量]`）：叠加 `count × 8` 小时伽马射线暴 buff（等价于同时拥有太阳风+流星雨+恒纪元），可叠加时长
- UTR自选券（`钓鱼使用 UTR自选券 鱼名`）：消耗 1 张兑换指定 UTR；条件为目标鱼所在地图已解锁至少 1 条 UTR
- 时光药水（`钓鱼使用 时光药水 [数量]`）：模拟8小时/瓶的钓鱼结算

- **两阶段结算**：时光药水使用时先结算历史待处理时间（阶段一），再结算药水时间（阶段二），避免用户迷惑（如2点开始钓鱼、5点使用时光药水，前3小时的鱼必须先结算）。
  - **阶段一（正常模式）**：`pending_minutes = now - last_settle_time`，若 > 0，以正常模式结算这段时间。使用历史时刻的 buff（不冻结），剩余不足一条鱼的时间按概率出鱼（小数不丢弃）。保底计数、猫吃鱼、猫礼物、流星鱼全部计入。
  - **阶段二（时间吸收模式）**：结算 `hours * 60` 分钟。使用当前时刻的 buff（`freeze_buff_time=now`），不足一条鱼的时间丢弃。仅此阶段过滤多多和幸运 buff。
  - 两阶段结果全部合并到 `status_dict`，但 `last_settle_time` 只推进到当前真实时间 `now`。药水的虚拟时间已经以鱼获形式写入状态，不能推进真实时间锚点，否则后续正常等待会因为 `last_settle_time` 在未来而不产鱼。鱼饵消耗即时扣减。用户收杆时由 `end_fishing` 统一处理所有鱼获。

- **削弱机制**：仅阶段二忽略多多药水和幸运药水 buff（它们与时光药水机制冲突），阶段一正常享受所有 buff

- **时间吸收架构（统一结算路径）**：时光药水与正常钓鱼走**同一条 `simulate_fishing_loop` 路径**，区别仅在于「时间从哪里来」：
  - 正常钓鱼（阶段一）：`current_time` 从 `settle_start` 推进到 `ctx.now`，每条鱼消耗 `fishing_interval` 分钟实际时间
  - 时光药水（阶段二）：`current_time` 不推进，每条鱼从 `time_credit_minutes`（= `hours * 60`）中扣除 `fishing_interval` 分钟。余额不足时丢弃剩余时间并结束
  - 引擎通过 `time_credit_minutes` 参数切换模式：`None` = 正常模式（时间推进），非 None = 时光药水模式（时间吸收）

- **鱼入状态架构**：时光药水**不做任何独立结算**——阶段一和阶段二的鱼获全部合并到钓鱼状态（`status_dict`），用户收杆时由 `end_fishing` 统一处理：入包、猫礼物发放、保底写回、自动锁鱼/卖鱼/卖猫乐园材料。

- 实现位置：`core/potion.py` 的 `use_time_potion_settle`，两阶段共用 `_prepare_fishing_context(gm_mode=False)`：阶段一用原始 `status_dict`（`settle_start` = 历史时间），不传 `freeze_buff_time`；阶段二设 `ctx_status_potion["last_settle_time"]` = `now`，过滤多多/幸运 buff 后传入 `simulate_fishing_loop(time_credit_minutes=hours*60, freeze_buff_time=now)`；阶段一的 bait_remaining 手动继承到阶段二

## 幸运药水

- 幸运药水（`钓鱼使用 幸运药水`）：8h 内双次结算取更高稀有度
- **对 UTR/木框/猫框也有效**：幸运激活时，木框/UTR/猫框参与"两次独立结算取最优"逻辑——任一次结算命中即掉落，而非单独调高概率
- **保底仅计算一次**：尽管是两次结算，保底计数器（frame_pity/utr_pity/cat_frame_pity）每次钓鱼只 +1
- **与真多多药水互斥**：同一时间只有 1 种药水可以生效，两者不可叠加使用

## 保底机制架构（三段式解耦）

`_catch_fish_with_buffs` 采用三段式结构，保底逻辑与随机结算完全解耦：

- **Part 1（保底阈值检查）**：在函数顶部检查保底计数器是否达到阈值，若满足则直接给予保底奖励并清零，跳过后续所有逻辑
- **Part 2（随机结算）**：执行随机概率拦截（木框0.7%/UTR0.2%）+ 稀有度表抽选。结算次数（1次或2次幸运）完全在此部分内处理，不触碰保底代码
- **Part 3（保底计数更新）**：根据 Part 2 的最终结果类型更新计数器——展示木框清零 frame_pity，迷途风UTR清零 utr_pity，普通鱼则两者 +1
- **猫猫乐园材料与保底**：`_catch_fish_at_interval` 中材料掉落同样记入保底计数（frame_pity +1，迷途风天气下 utr_pity +1，与普通鱼一致）；但**保底优先于材料**——若展示木框或迷途风UTR保底即将触发（`(计数+1) >= 阈值`），跳过材料掉落直接进入钓鱼结算，由 Part 1 处理保底奖励

猫猫框保底在 `core/cat.py` 的 `process_cat_gift` 中同样采用此结构：保底检查 → 随机结算 → 计数更新。触发条件为每次被猫吃掉鱼但未掉落框时 +1（获得框时归零）

## UTR 机制详解

### UTR 获取途径（按地图类型）

| 地图类型 | UTR 来源 | 解锁条件 | 概率机制 | 保底 |
|---------|---------|---------|---------|------|
| 1-10 图 | **集齐 UR + 迷途风**（双重要求） | 集齐该图所有 UR → 解锁迷途风可见/生效；仍须在迷途风天钓鱼 | 递进概率 `0.2% + lead×0.1%` | 150 次（仅迷途风天气时计数） |
| S1 猫猫乐园 | **集齐 UR + 迷途风**（双重要求） | 同上 | 同上 | 同上 |
| 11-20 星空图 | 解锁后常驻递进判定（不依赖迷途风） | 集齐该图所有 UR → max_rarity 从 UR→UTR | 递进概率 `0.2% + lead×0.1%`（概率表 UTR 截断为 0） | 150 次（UTR 解锁后任意天气计数；未解锁不计） |

### 1-10 图与 S1 的 UTR 机制

> **注意**：帮助文案不得写成「仅依赖迷途风」。玩家既要收集该图全部 UR，又要等待迷途风天气。

1. 玩家集齐当前图所有鱼的 N/R/SR/SSR/UR → 成就 `collect_scene_{id}` 完成
2. `achievement_service.py` 显示"🌀 {location.name}的迷途风天气已对你解锁！"
3. `has_unlocked_lost_wind(user_id, location_id)` 返回 True → 迷途风天气 buff 不再被过滤
4. 当迷途风天气生效时（`weather_lost_wind=True`）：
   - **递进概率**：`get_lost_wind_utr_probability(rod_level, difficulty)` = `0.002 + max(0, rod_level - (difficulty+1)) × 0.001`
   - **150 保底**：`utr_pity + 1 >= UTR_PITY_THRESHOLD(150)` 时必定获得 UTR
   - **计数器递增**：每次钓鱼 `utr_pity += 1`（仅迷途风天气时）
5. 无迷途风天气时：递进概率不调用、保底计数器不递增、不出 UTR
6. 未集齐 UR 时：即使地图实际为迷途风，对该玩家也视为未解锁（预报显示晴天、不生效）

### 11-20 星空图的 UTR 机制

1. 玩家集齐当前星空图所有鱼的 N/R/SR/SSR/UR → 成就 `collect_scene_{id}` 完成
2. `achievement_service.py` 显示"✨ {location.name}的UTR稀有度已对你解锁！\n解锁后递进概率与 150 次 UTR 保底常驻生效（星空图不生成迷途风）。"
3. `core/engine.py` / `core/scene.py` / `core/actions.py`：运行时 `max_rarity` 从 UR → UTR；`utr_special_active = (is_starry and max_rarity == "UTR")`
4. **不依赖迷途风**：11-20 **不会生成迷途风天气**；解锁后递进概率与 150 保底常驻，与太阳风/流星雨/恒纪元无关
5. UTR 解锁后（`max_rarity == "UTR"`）：
   - **递进概率**：`get_lost_wind_utr_probability(rod_level, difficulty)` 生效（d=6 时约 0.7%）
   - **150 保底**：`utr_pity + 1 >= 150` 时必定获得 UTR；每次钓鱼 `utr_pity += 1`（出 UTR 归零）
   - **概率表 UTR**：星空图概率表 index≥5 仍强制截断为 0，只走递进判定
6. UTR 未解锁时（`max_rarity` 被压到 `"UR"`）：
   - 递进概率不启用、保底计数器不递增
   - 星空专属天气只影响流星鱼，不改 UTR 规则

### 概率表 UTR 截断问题

稀有度分布采用高斯分布 + 尾部裁剪（`THETA=0.05`）：概率低于 5% 的项被截断为 0 并重新分配。此外，**星空图（11-20）中概率表的 UTR（index>=5）被代码强制截断为 0**，UTR 仅可通过解锁后的递进概率和 150 保底获得（数值公式与 1-10 迷途风一致，但不要求迷途风天气）。

实现：`core/engine.py` 的 `_single_random_roll` 中，`is_starry=True` 时将 `probabilities[5:]` 全部置 0 并将概率质量重分配给 `probabilities[4]`（UR），在 `apply_meteor_effect` 之前执行。

| d 值 | 杆等级 | UTR(截断前) | UTR(截断后) | 星空图实际UTR | 说明 |
|------|--------|------------|------------|-------------|------|
| 6 | Lv.16 | 0.25% | 0.00% | 0.00% | 概率表截断，解锁后递进约 0.7% |
| 9 | Lv.19 | 10.78% | 11.34% | **0.00%** | 星空图强制截断，解锁后递进约 1.0% |
| 10 | Lv.20 | 22.83% | 25.07% | **0.00%** | 星空图强制截断，解锁后递进约 1.1% |

- 星空图 UTR 唯一来源：集齐全 UR 解锁后的递进概率 + 150 保底（与天气无关）
- 星空专属天气（太阳风/流星雨/恒纪元）只改流星鱼；概率表 UTR 仍截断为 0
- 1-10 图：概率表本身无 UTR 概率（截断算法 THETA=0.05），UTR 仅来自迷途风递进 + 保底

### 递进概率算法

`constants.py` 的 `get_lost_wind_utr_probability(rod_level, location_difficulty)`：

```
location_level = location_difficulty + 1
lead = max(0, rod_level - location_level)
probability = 0.002 + lead × 0.001  # 即 0.2% + lead × 0.1%
```

图 11（difficulty=10）示例：

| d 值 | 杆等级 | lead | 递进 UTR 概率 | 概率表 UTR | 星空图实际生效？ |
|------|--------|------|-------------|-----------|-----------|
| 6 | Lv.16 | 5 | **0.70%** | 0.00% | ✓ 解锁后常驻递进 0.7%+保底 |
| 7 | Lv.17 | 6 | 0.80% | 0.00% | ✓ 解锁后常驻递进 0.8%+保底 |
| 8 | Lv.18 | 7 | 0.90% | 0.00% | ✓ 迷途风日递进0.9%+保底 |
| 9 | Lv.19 | 8 | 1.00% | 11.34% | ✓ 迷途风日递进1.0%+保底（概率表被截断为0） |
| 10 | Lv.20 | 9 | 1.10% | 25.07% | ✓ 迷途风日递进1.1%+保底（概率表被截断为0） |

- 递进概率算法仅在 `weather_lost_wind=True` 时由 `engine.py` 调用
- 星空图50%迷途风日 → 递进概率和150保底在迷途风日生效
- 星空图50%流星日 → 概率表UTR被截断为0，流星+2%加成作用于UR

### 150 保底机制

- `UTR_PITY_THRESHOLD = 150`（`constants.py`）
- 保底检查（`engine.py` Part 1）：`if weather_lost_wind and (utr_pity + 1) >= 150`
- 计数器更新（`engine.py` Part 3）：`new_utr_pity = utr_pity + 1 if weather_lost_wind else utr_pity`
- **迷途风日**（50%概率）：计数器递增，保底可触发
- **流星日**（50%概率）：计数器不递增，保底不触发

## 黑商/白商交换

- 指令：`黑商[交换] 鱼名字稀有度 鱼名字稀有度`，别名 `黑市`；入口只做前缀匹配，参数在 handler 中按纯文本解析；也支持鱼名与稀有度空格分隔或紧贴，如 `黑商 鲤鱼UR草鱼SSR`、`黑商交换 鲤鱼 UR 草鱼 SSR`
- 交换参数宽松解析：检测到两段鱼名+稀有度，或两组 3 位鱼 ID（如 `黑商 123 456`）才进入解析；单独输入 `黑商`/`黑市` 回复用法提示，不像交换的黑/白商前缀消息仍静默不回复并写入日志
- 解析后若发现来源鱼不在背包而目标鱼在背包，会自动交换来源/目标变量，支持用户反向输入
- 黑商条件：玩家背包有来源鱼；来源鱼场景等级 ≥ 目标鱼场景等级，来源鱼稀有度 ≥ 目标鱼稀有度，允许场景等级与稀有度同时相等
- 场景等级按 `locations.json` 的 `difficulty + 1` 计算，S1 猫猫乐园 difficulty=6 视为 7 级图
- 锁定鱼也允许被交换消耗；黑商每天仅可交换 1 次
- 黑商相同稀有度交换时，70% 概率改为目标鱼所在地图、相同稀有度的其他鱼；随机结果不能是目标鱼或消耗鱼，若无其他鱼则仍给指定目标
- **黑商秘密保底**：连续4次被随机替换目标鱼（"失败"）后，下次必定获得玩家指定的目标鱼，不触发随机替换。保底计数器记录在 `FishingUser.black_market_pity_counter` 字段，获得指定目标时清零，被随机替换时+1。不向玩家显示任何提示
- 黑商成功后写入 `FishingExchangeRecord`，记录交换人、消耗鱼与实际获得鱼；若触发随机，记录实际获得鱼；白商会过滤交换前后编号相同的旧记录
- 指令：`白商`，别名 `白市`；展示有效黑商记录的反向交换列表，按二区分类并使用二重分组压缩显示：
  - **现在可交换**：用户背包已有 source 鱼（可直接发起白商交换）
  - **有可能做到**：用户已解锁 source 鱼所在地图且鱼竿等级足够钓到该稀有度（UTR 鱼还需 `collect_scene_{id}` 成就）
  - 未解锁地图、鱼竿等级不足以钓到该稀有度、或缺少 UTR 成就的记录均不展示
  - 地图解锁规则：普通图(1-10)看 rod_level ≥ difficulty；星空图(11-20)需已建星空艇 + rod_level ≥ difficulty；S1 需猫猫乐园门票（已解锁任意 2 组普通图 UTR 全收集）
  - 鱼力检查：非 UTR 鱼额外检查 `get_rarity_probabilities(rod_level, difficulty)` 中该稀有度概率 > 0，避免已解锁地图但鱼力不足以钓到 UR/SSR 的记录出现
  - UTR 前置：`collect_scene_{id}` 要求该图所有鱼 × {N,R,SR,SSR,UR} 全收集，隐含"已收集到该图所有 UR"，未集齐则 UTR 不展示
  - **二重分组**：第一重按左侧相同的鱼合并（只显示一次左侧鱼），第二重按右侧相同场景合并（只显示一次场景名）；左侧鱼按稀有度降序→场景等级升序→名称排序，右侧场景按场景等级升序排序，右侧鱼按稀有度降序→名称排序
  - 相同反向交换（白商交出鱼与获得鱼均相同）只显示排序靠前的第一条，不显示黑商达成者昵称
  - 页面顶部说明：因为有人用右边的鱼找黑商换过左边的鱼，所以你可以使用左边的鱼交换右边的鱼
- 指令：`白商[交换] 鱼名字稀有度 鱼名字稀有度`，别名 `白市`；白商列表仍使用无参数 `白商` / `白市`；对于每条黑商记录，任何玩家都可以进行反向交换；交换百分百成功且无随机性，成功后对应黑商记录失效，并显示是谁帮助了你
- 交换记录表 `FishingExchangeRecord` 持久化记录创建时间 `create_time`、更新时间 `update_time` 和有效状态 `is_active`；白商交换成功后消耗 1 次赠送次数，并将对应记录置为失效，不再展示或可交换
- 交换获得鱼走 `add_fish_to_user()`，因此 UTR 首获会自动消耗解锁图鉴

## 锁鱼/解锁通配符语法

- `锁鱼`/`解锁` 支持以下选择器，可用逗号/分号/空格批量混用：
  - 鱼ID（如 `锁鱼 111`）
  - 稀有度字母（如 `锁鱼 SR`、`锁鱼 UTR`）
  - 稀有度索引（如 `锁鱼 **3` = SR，`**6` = UTR）
  - 位置通配符（如 `锁鱼 S1**` 锁定猫猫乐园全部鱼；`锁鱼 1**` 锁定地图1全部鱼）
  - 全部（`锁鱼 全部`）
- 位置通配符格式为 `位置ID**`，匹配鱼ID以该位置前缀开头的所有鱼。S1 猫猫乐园鱼的ID前缀为 `s1`（小写），普通图N的ID前缀为数字N
- 实现链：`backpack/selection.py` 的 `parse_fish_selection` 将 `前缀**` 解析为 `location_prefixes` → `backpack/lock.py` 的 `_toggle_fish_lock` 调用 `FishingUser.lock_by_location_prefix` / `unlock_by_location_prefix`（`models/user.py`）按 `numeric_id.startswith(prefix)` 过滤

## 锁鱼/解锁误触过滤

- `锁鱼`/`解锁` 指令的正则 `锁鱼(?:\s*(.+))?` / `解锁(?:\s*(.+))?` 会匹配以"锁鱼"/"解锁"开头的任意文本（如群聊"解锁道具组合是这个游戏充值服务爽点"），导致误回复"未找到编号为…的鱼"
- 过滤策略：在 handler 层（非正则层）加校验，正则不变，不影响多鱼种批量匹配功能
- 判定函数 `is_likely_misfire(fish_input)`（`backpack/selection.py`）：输入含汉字且不含任何有效选择器（rarity字母 / **N / 前缀** / 全部|all|所有 / 纯数字ID）时返回 True
- 命中误触时静默不响应（handler 直接 return），不回复任何内容
- 仅作用于 `锁鱼` 和 `解锁` 两个指令

## 自动锁鱼

- `自动锁鱼` 指令：收杆后将自动锁定匹配的鱼，优先级**高于**自动卖鱼（先锁后卖）
- 指令格式：
  - `自动锁鱼`：查看当前状态
  - `自动锁鱼 开启`：开启（使用已保存的通配符）
  - `自动锁鱼 关闭`：关闭
  - `自动锁鱼 S1**`：设置通配符并开启（支持所有锁鱼通配符：鱼ID、稀有度SR、稀有度索引**3、位置通配符S1**/1**、全部）
- 数据模型：`FishingUser.auto_lock`（BooleanField）+ `auto_lock_pattern`（CharField，存通配符表达式）
- 实现链：`handlers/fishing.py` 收杆时检查 `get_auto_lock` → `auto_lock_fish(user_id, pattern)`（`backpack/lock.py`）→ 再执行自动卖鱼；时光药水只把鱼获合并进钓鱼状态，自动锁/卖在后续收杆时统一触发
- 结算页面：仅在 hints 中标注 `🔒 自动锁鱼：已锁定 N 种`，不显示详细鱼名
- Web 端：`api.py` 返回 `auto_lock_enabled` / `auto_lock_pattern`；`index.html` 显示自动锁鱼状态；`command_router.py` 注册 `自动锁鱼` 路由

## UTR 卖鱼保护

- `卖鱼`（不加参数）默认卖出全部鱼但**排除 UTR 稀有度**，防止误卖
- `卖鱼 UTR`、`卖鱼 全部`、`卖鱼 **5` 等显式指定参数时正常处理，UTR 鱼可正常卖出
- 自动卖鱼不受影响，按设定稀有度正常卖出
- 实现：`sell_fish()` 增加 `exclude_utr` 参数，`_get_sellable_fish()` 在 `select_all` 且 `exclude_utr=True` 时过滤 UTR

## GM 全服模式

以下指令支持「全服」模式：将 `@用户` 替换为 `全服` 即可对全服所有用户执行相同操作。未 @用户 也未声明全服时，将提示需要指定目标。

- `gm添加 物品名 数量 全服`：全服添加物品（鱼、鱼饵、药水、展示框、猫猫框、流星鱼等）；数量为负数时扣除，余额不足则失败

## 药水系统

| 药水 | 效果 | 持续 | 使用方式 | 实现入口 |
|------|------|------|---------|---------|
| 时光药水 | 每瓶模拟8小时钓鱼，不受其他药水的影响 | — | `钓鱼使用 时光药水 [数量]` | `core/potion.py` → `use_time_potion_settle` |
| 真多多药水 | 鱼竿等级-1，钓到的鱼数量翻倍；对流星鱼也生效（同编号掉两条）；不可叠加，重复或一次使用多个会延长时间 | 8h/瓶 | `钓鱼使用 真多多药水 [数量]` | `shop/potion_use.py` → `use_duoduo_potion` |
| 回档药水 | 清空当前钓鱼进度（鱼获归零，重新结算） | — | `钓鱼使用 回档药水` | `shop/potion_use.py` → `use_rollback_potion` |
| 幸运药水 | 双次结算取更高稀有度，同稀有度优先图鉴未收集鱼种；不可叠加，重复使用时间往后堆叠（剩余+8h） | 8h | `钓鱼使用 幸运药水` | `shop/potion_use.py` → `use_lucky_potion` |

- 真多多药水机制：单实例 buff（`BUFF_TYPE_DUODUO`），使用时若已存在未过期多多 buff 则 `end_time += 8h * 数量`，否则新建 `8h * 数量` buff；效果不可叠加，固定 `rod_level -= 1`，`quantity = 2`；**数量翻倍对 UTR 鱼生效，但对展示木框等非鱼类道具不生效**（保底木框路径返回 quantity=1）；**对流星鱼后置生效**：不改掉落率与编号生成，在 `roll_starry_fish` 得到最终编号后由 `expand_starry_fish_with_duoduo` 检查多多并复制为两条**相同编号**流星鱼（展示/入库/抽奖各计 2 条）；**多多药水支持小数倍率合并**：`_compute_duoduo_quantity(duoduo_count, cat_park_double_rate)` 将多多整数倍率（`1 << duoduo_count`）与猫乐园旋转逗猫棒的小数概率倍率合并为浮点总倍率，整数部分为保底数量，小数部分为额外 +1 的概率（如 1.10 → 90%得1条/10%得2条），使两种翻倍机制在结算层面统一
- 幸运药水机制：单实例 buff（`BUFF_TYPE_LUCKY_DOUBLE`），使用时若已存在未过期幸运 buff 则 `end_time += 8h`，否则新建 8h buff；引擎 `_catch_fish_with_buffs` 中抽第二条鱼，按封顶后稀有度比较——更高稀有度取第二条，相同稀有度时若第一条已图鉴收集而第二条未收集则取第二条；钓鱼状态页左上角显示 ⭐ 图标
- **真多多/幸运药水互斥**：同一时间只能有 1 种药水生效。使用幸运药水前若检测到未过期的真多多 buff（`BUFF_TYPE_DUODUO`）则拒绝使用；使用真多多药水前若检测到未过期的幸运 buff（`BUFF_TYPE_LUCKY_DOUBLE`）则拒绝使用；提示"同一时间只有1种药水可以生效"。检查发生在扣减药水之前，未消耗道具
- 所有药水（许愿、真多多、回档、幸运、时光等）均不在鱼店出售：`buy_item` 对 `item_type="potion"` 直接拒绝购买，不扣费；通过摩天轮签到、活动或 GM命令（`gm添加 物品名 数量 @用户`）获取，使用走 `钓鱼使用 物品名`
- 背包道具区必须从 `FishingUser.items` 汇总所有 `item_type="potion"` 且数量大于 0 的药水展示；兼容历史 `time_potion` key 显示为时光药水，真多多药水和幸运药水按配置名显示。
- 药水使用分发表：`shop/item_dispatch.py` → `ITEM_DISPATCH`

### 流星鱼系统 / 星空祈愿（技术实现细节）

> **以下内容为技术实现细节，仅供开发参考，不得在 help.html 等玩家可见页面展示。**

#### 基础参数

- 活动【星空祈愿】的活动鱼为**流星鱼**：在 11-20 图钓鱼时基础 **5% 概率掉落**；鱼竿等级超过 10 后每高 1 级额外 **+0.5%**（绝对加值，如 Lv.11=5.5%、Lv.20=10%）。每条流星鱼拥有一个 **6 位数字 ID**（000000-999999）。
- 流星鱼按 6 位数字番型评分（32 种番型，使用 -log10(概率) 加权），分数对应奖池：0 分无奖励、1-2 分低级、3-5 分中级、6-10 分高级、11 分及以上究极。对子家族含「两对」「三对」：统计恰好长度为 2 的同号连段（更长同号连段不计对），三对优先于两对且不重复计分。另有「葫芦」：任意 5 位窗口为 AAABB 或 AABBB，存在即计一次。
- 流星鱼奖池列表：低级=玉米/黑商额外兑换券/中级抽奖碎片/0.5积分（等概率，0.5积分会额外推进星空祈愿达成）；中级=多多药水/幸运药水/重置药水/猫框/高级抽奖碎片；高级=闪光药水/时光药水/UTR 自选券/究极抽奖碎片；究极=时光药水×10/UTR 自选券×10。
- 抽奖碎片永远比当前奖池高一级（低级池→中级碎片，中级池→高级碎片，高级池→究极碎片）；不存在“低级抽奖碎片”这种产出。
- 底层规则集中在 `core/starry_system.py`：包含 ID 生成、评分、掉落率（`get_starry_fish_drop_rate`）、流星鱼比较、天气/幸运药水抽取和奇迹子集算法。
- 玩家流星鱼独立存储在 `FishingUser.starry_fish`，4158 `/starry` 会暴露累计分数、流星鱼数量、规则阈值、S2 入场券来源和奖池边界/内容。
- 碎片自动升级链（本杆立即结算，可连锁）：5 中级抽奖碎片→中级奖池、5 高级抽奖碎片→高级奖池、5 究极抽奖碎片→究极奖池。S2 入场券只通过累积流星鱼评分获得。
- 流星鱼**无法赠送、卖出、锁定**。

#### S2 入场券：累计评分进度

- **阈值**：调整为整数目标 **1200 分**（便于玩家理解；原模拟推导为 70 天中位数 × 130% ≈ 1129.7 分）。
- 玩家累积流星鱼分数达到 **1200** 分后获得 S2 入场券。收杆页仅在本杆钓到流星鱼时显示进度条与累计分/目标分，不展示「努力值」「还差」等文案。
- 此机制替代了原有的随机掉落方案，降低运气方差，保证努力型玩家一定能获得。

#### 新天气（11-20 图）

- 每天 **5 张**图有额外星空天气（太阳风 / 流星雨 / 恒纪元），另 **5 张**为乱纪元；天气每天 11 点刷新。不生成迷途风。
- **太阳风**：流星鱼出现率 **恒定 +2.5%**（绝对加值，不与鱼竿等级加成乘算叠加）。
- **流星雨**：星空鱼变得幸运。
- **恒纪元**：流星鱼数字仅包含 **2-8**（不出现 0、1、9），共 7^6=117649 种组合；此规则不在游戏内体现，仅文档说明。
- **乱纪元**：无额外效果。
- **闪光药水**：激活伽马射线暴 buff（持续 8 小时），同时触发太阳风 + 流星雨 + 恒纪元三种天气效果。

#### 奇迹功能

- **触发条件**：仅**背包**（`starry_fish`）内流星鱼 ID 子集之和尾号为 **7777777**（即子集和 `mod 10_000_000 == 7_777_777`）。**展馆鱼不参加奇迹运算**。
- 触发后消耗对应背包子集，获得**星辰木框**；收杆页以图示标出奇迹要因编号。展馆鱼不参加运算、也不会被消耗。
- **算法实现**：使用 meet-in-the-middle（二分法）：把候选鱼拆成两半，分别枚举子集和取模，用排序/二分匹配目标余数，避免 2^n 全量暴力。

#### 星辰木框与星空木框

- **星辰木框**（`star_frames`）：奇迹奖励，背包「重要」区显示；上限见 `STAR_FRAMES_MAX`。
- **星空木框**（`starry_frames`）：建设星空艇后可升级；消耗**星辰木框**（`star_frames`）按 `STARRY_FRAME_UPGRADE_COSTS`（1+2+…+10=55）升级，最多 `STARRY_FRAMES_MAX=10` 个。
- **展示收益倍率**（按展示鱼售价从高到低）：前 `starry_frames` 条 **4×**，其后至 `upgraded_display_count` 为猫猫框 **3×**，其余普通展示木框 **2×**（`services/display_service.py`）。
- 指令：`升级展示栏` / `增加展示栏位` / `强化展示栏位` / `升级星空木框` / `星空木框` 均为万能升级入口，走 `upgrade_display_slots`：
  - 木框栏位 / 猫猫框强化 / 星空木框 未满 10 的加入校验列表；
  - 材料足够的全部升级并扣框；
  - 若一个都升不了，汇总报告各自还差多少。
- 背包：重要区展示星辰木框、星空木框数量；展示区最贵的 `starry_frames` 条鱼套用星空框样式。

#### 流星鱼展馆（星空祈愿）

- 分数 ≥ 4 分的流星鱼自动进入个人展馆（最多 10 条），通过 `流星鱼展馆` / `星空祈愿展馆` / `星空鱼展馆` 命令查看。

- **计分/努力值不消耗流星鱼**；只有奇迹触发时才会消耗**背包**中的对应子集，展馆鱼不参加奇迹运算，也不会被消耗。
- 收杆结算页会把本次获得的流星鱼放入独立区块，不再混入普通鱼获格子。

## 时间敏感型Buff检查原则

**问题：** 钓鱼模拟是回溯时间线（从开始时间到当前时间按时序推进）。每次捕获发生在特定的模拟时间点。如果 Buff 检查只在模拟循环开始或结束时做一次，而不是按每次捕获的模拟时间点检查，会导致活动区间外的捕获也被算入。

**正确做法：**
1. 在循环开始前从 `ctx.buffs` 提取出该事件 buff 列表（非布尔值）
2. 每次捕获时，用当前模拟时间 `catch_time` 检查 buff 时间区间：`any(b.start_time <= catch_time <= b.end_time for b in event_buffs)`
3. 数值型 buff 已经在 `get_effects_at_time(ctx.now)` 中按时间点正确叠加，不存在此问题

4. **时区陷阱（重要！）**：Tortoise-ORM 配置了 `timezone: "Asia/Shanghai"`，所有 `DatetimeField` 从数据库读出都是 **timezone-aware** 的。而 `datetime.now()` 是 timezone-**naive** 的。两者直接比较会抛出 `can't compare offset-naive and offset-aware datetimes`。
   - **防护函数**：`_make_naive(dt)`（`models/user.py`）直接 `replace(tzinfo=None)`；`_to_local_naive(dt)`（`render/fishing_status.py`）先 `astimezone()` 再剥离，但对远期日期（如永久 buff 的 2099 年）会降级为 `replace(tzinfo=None)`（Windows 时间戳上限约 3001 年，`astimezone()` 对更远日期抛 `OSError`，捕获后直接剥离即可，因该值会被 `min(buff_end, local_end)` 截断）
   - **规则**：任何涉及 DB datetime 与 `datetime.now()` / 模拟时间 `catch_time` 的比较，**必须先对 DB 时间调用 `_make_naive()`**
   - **永久 buff end_time 年份上限**：永久/远期 buff 不得使用 9999 年，统一用 `STARRY_BONUS_END_YEAR = 2099`（或更小），避免触发 Windows `astimezone()` 的 `OSError: [Errno 22]`
   - **已出事故**：端午活动 Buff 的时间区间检查（`engine.py`）直接用 DB 的 `b.start_time/end_time` 与 naive `catch_time` 比较，导致 2026-06-20 钓鱼状态渲染崩溃

5. **Buff 元数据注册表（重要！）**：所有 `buff_type` 必须在 `BuffEffect.BUFF_REGISTRY` 中注册 `BuffMeta`（含 `key`/`display_name`/`color`/`emoji`），否则显示层无法获取中文名，会回退到英文 buff_type 字符串暴露给玩家。
   - **单一数据源**：`render/fishing_status.py` 的 `BUFF_TIMELINE_COLORS` 和 `services/buff_service.py` 的 `generate_buff_message` 均从 `BUFF_REGISTRY` 派生，不再各自维护独立映射字典
   - **新建 buff 流程**：① 在 `BuffEffect` 类中定义 `BUFF_TYPE_XXX` 常量 → ② 在 `BUFF_REGISTRY` 中添加 `BuffMeta` 条目（必须配置中文名 `display_name` 和颜色 `color`）→ ③ 如需参与钓鱼计算，在 `BUFF_EFFECTS` 中添加效果映射
   - **类方法**：`BuffEffect.get_meta(bt)` 返回 `BuffMeta | None`；`BuffEffect.get_display_name(bt)` 返回中文名（未注册回退英文）；`BuffEffect.get_color(bt)` 返回颜色（未注册回退灰色）
   - **已出事故**：`lucky_double` 和 `dragon_boat` 未在 `BUFF_TIMELINE_COLORS` 中配置，时间轴图例直接显示英文 `lucky_double`/`dragon_boat` 给玩家

> 赠送 UTR 鱼若触发对方首次 UTR 图鉴解锁并被自动消耗，不计入赠送者每日赠送次数；若对方已解锁并实际入包，则正常计次。

> 赠送 UTR 鱼触发他人首次 UTR 图鉴解锁时，赠送者获得该 UTR 鱼 1 倍价格的金币奖励。

## 测试与 Mock 指南

### 项目基于 nonebot 的测试约束

本插件基于 nonebot2 框架，但 nonebot 的 `require()` 机制和插件加载器会在 import 时触发框架初始化，直接 `pytest` 无法导入插件模块。因此所有测试需要在导入前 mock 掉 nonebot 及其生态依赖：

1. **Meta-path finder 方案**：使用 `sys.meta_path` 插入自定义 finder，对 `nonebot`、`tortoise`、`arclet` 等包前缀自动生成 MagicMock 模块。这比手动逐个 `sys.modules[name] = MagicMock()` 更健壮，能处理深层子模块（如 `nonebot.internal.matcher`）的自动导入
2. **预创建顶层包**：`nonebot`、`tortoise`、`arclet` 需要预创建为带 `__path__` 的 MagicMock，否则 `import nonebot.plugin` 时代码会认为不是包而报错
3. **nonebot_plugin_* 系列**：`nonebot_plugin_apscheduler`、`nonebot_plugin_session`、`nonebot_plugin_htmlrender` 等需要单独 mock；`html_to_pic` 需设为 `AsyncMock(return_value=b"FAKE")`
4. **zhenxun 子模块**：`zhenxun.services.log`、`zhenxun.configs.utils`、`zhenxun.models.user_console` 等需要 mock，但**不要 mock `zhenxun` 本身**（需要真实包让 `zhenxun.plugins.*` 的 `__init__.py` 正常执行）
5. **FishingUser 类方法**：引擎在 `simulate_fishing_loop` 中调用 `FishingUser.get_user_fish`、`get_user_collected`、`get_status`、`get_user_items`、`get_item` 等异步类方法，测试前需要设置 `FishingUser.xxx = AsyncMock(return_value=...)`

### Mock 心得

- **AutoMockFinder 模式**：定义 `find_spec(name, path, target)` 返回 `ModuleSpec`，`create_module(spec)` 返回带 `__path__` 的 MagicMock，`exec_module(module)` 空实现。这样 Python 导入系统会自动调用这些方法创建 mock 模块，无需预知所有子模块名
- **MagicMock 的属性穿透**：`MagicMock()` 会自动为任意属性访问返回新的 MagicMock，但**不会**为 `hasattr` 返回 False 的属性自动创建。如果代码用 `getattr(obj, "attr", default)` 模式访问，MagicMock 会返回 Mock 对象而非 default，可能改变代码行为
- **AsyncMock vs MagicMock**：所有被 `await` 调用的方法必须用 `AsyncMock`，否则 `await` 会抛 `TypeError: object MagicMock can't be used in 'await' expression`
- **导入顺序**：mock 必须在 `from zhenxun.plugins...` 之前完成。一旦真实模块被导入，`sys.modules` 中的 mock 会被覆盖
- **测试脚本位置**：独立测试脚本放在临时工作目录（`c:\Users\Administrator\.trae-cn\work\...`），不要污染插件目录；正式 pytest 测试放在 `tests/` 目录，参考 `test_config_and_logic.py` 的 mock 模式

### 可复用的 AutoMockFinder

```python
class AutoMockFinder:
    def __init__(self, prefixes):
        self.prefixes = prefixes
    def find_spec(self, name, path, target=None):
        for prefix in self.prefixes:
            if name == prefix or name.startswith(prefix + "."):
                from importlib.machinery import ModuleSpec
                spec = ModuleSpec(name, self)
                spec.submodule_search_locations = []
                return spec
        return None
    def create_module(self, spec):
        mod = MagicMock()
        mod.__name__ = spec.name
        mod.__loader__ = self
        mod.__package__ = spec.name
        mod.__spec__ = spec
        mod.__file__ = f"<auto_mock:{spec.name}>"
        mod.__path__ = []
        return mod
    def exec_module(self, module):
        pass

# 使用：对需要 mock 的包前缀注册
finder = AutoMockFinder(["nonebot", "tortoise", "arclet", "nonebot_plugin_htmlrender"])
sys.meta_path.insert(0, finder)
```
