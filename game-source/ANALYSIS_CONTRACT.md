# 游戏源码分析契约

本契约是“分析游戏源码”任务的唯一详细流程。`AGENTS.md` 只保留入口级硬约束；本文件负责具体步骤、报告格式、验证和失败恢复规则。

## 触发与授权边界

- 用户要求“分析游戏源码”或提出等价请求时，即视为要求分析最新上游版本；不要求用户额外说明“新版”或“更新”。
- 源码分析是分析任务，不包含利润网修复授权。必须只读检查利润网实现、测试、配置、PRD 和生成产物，不得顺手修复，也不得为了让测试通过而更新源码哈希、快照或期望值。
- 分析阶段在主仓库内只允许产生以下变更：
  - 新增或更新本次分析报告。
  - 维护报告索引或当前固定 SHA 说明。
  - 在全部分析与验证完成后更新 `game-source/current` 的 gitlink。
- 除上述文件外，不得修改利润网代码、测试、配置、PRD、样式或生成产物。
- 完成分析、验证、报告和允许的 gitlink/索引更新后必须停止，等待用户判断。只有用户随后明确要求修复、同步或更新利润网，才能进入独立的实现阶段。

## 上游获取与固定范围

- `game-source/current/` 是上游仓库 [`RShock/zhenxun_plugin_fishing`](https://github.com/RShock/zhenxun_plugin_fishing) 的 Git 子模块。利润网主仓库只记录经过确认的固定 commit，不归档源码压缩包，也不直接提交上游源码文件。
- 开始分析前记录子模块当前完整 SHA，称为“旧 SHA”。
- 在子模块中执行 `git fetch origin main`，把最新 `origin/main` 的完整 SHA 记录为“目标 SHA”。不得用 `git pull` 或 `git submodule update --remote` 代替。
- 记录旧 SHA 与目标 SHA 的提交关系、commit 列表和文件统计，并完整检查 `旧 SHA..目标 SHA` 的源码差异。
- 日常拉取利润网只能同步主仓库已记录的子模块 commit，不得自动跟随上游 `main`。
- 不得直接修改、提交或向上游推送子模块源码。

## 审查范围

- 必须采用“增量审查 + 整体闭环复核”。
- 增量审查必须覆盖 `旧 SHA..目标 SHA` 的全部变更，不得只挑看名称疑似影响利润的文件。
- 整体闭环必须从开始钓鱼、模拟捕获、鱼饵消耗、天气与 Buff、停止结算、奖励入库一路复核到出售定价。
- 无需无差别逐文件重读上游仓库，但必须覆盖所有影响利润的入口、配置和下游消费者，并确认变更是否改变鱼价、概率、数量、成本、奖励或状态推进。
- 必须同步只读检查利润网实现、测试、配置、PRD 和生成产物，判断它们与目标源码实际结算行为是否一致。
- 判断收益规则时以实际执行链和最终入库/出售结果为准；帮助文档、展示概率和测试期望只能作为证据，不能覆盖与其冲突的实际结算行为。

## 报告忽略规则

- 忽略规则只用于过滤“源码与利润网有差异但无法确定是否为 Bug”中的已确认产品边界。分析时仍须检查相关源码以完成闭环，但命中以下规则的差异不得写入报告：
  1. **钓鱼稳定收益边界**：利润网只计算钓鱼本身可长期重复的稳定收益，不计算依赖赠送或其他玩家状态的变现渠道；UTR 解锁型赠送奖励不属于利润。
  2. **长期稳态而非账号回放**：利润网不承担库存或逐账号状态模拟，按鱼饵可无限购买、选定状态稳定持续 24 小时的口径计算。因此不考虑有限鱼饵库存、耗尽与自动换饵、时光药水阶段库存、首条 UTR 被图鉴消耗、当前 UTR/展示木框保底计数或一次性成就奖励。
- 忽略规则不得掩盖可证明的游戏源码 Bug，也不得掩盖利润网在自身既定稳态口径内的计算错误；这两类问题仍分别写入“游戏源码 Bug”或“利润网 Bug”。
- 用户以后新增忽略规则时，必须先判断其语义是否与现有规则相同、相近、重叠或可被现有上位概念包含：
  - 可以合并时，扩充现有规则的定义或示例，不新增平级规则。
  - 只有无法在不改变原规则含义的前提下归并时，才新增一条规则。
  - 具体玩法、道具或状态优先作为既有规则的示例，不为每个实例单独建立规则。

## 报告格式

- 报告开头只记录审查范围：旧 SHA、目标 SHA 和固定 compare 链接。
- 除上述范围元数据外，所有发现必须且只能位于以下三个固定标题下：

  ```markdown
  ## 游戏源码 Bug
  ## 利润网 Bug
  ## 源码与利润网有差异但无法确定是否为 Bug
  ```

- 不得另设摘要、结论、建议、验证或其他承载实质结论的章节；某类没有发现时写“未发现”。内部的增量审查和收益闭环过程不展开成第四类或流水账。
- **游戏源码 Bug**：游戏源码内部存在可由实际执行链、数据流、承诺行为或自相矛盾结果证明的问题。
- **利润网 Bug**：利润网行为可被证明与目标源码的实际结算结果不一致。只报告，不在分析阶段修复。
- **源码与利润网有差异但无法确定是否为 Bug**：是否需要对齐取决于产品范围、估值口径或设计意图，必须写清需要用户决定的具体口径。
- 每项发现必须写明代码证据、影响范围和判定理由。证据不足或设计意图不明确的项目不得归入前两类；命中报告忽略规则以及未变化且无问题的参数无需写入报告。
- 上游证据必须使用包含完整目标 SHA 的固定链接，不得把会随子模块更新而漂移的 `game-source/current/` 路径用作长期证据。
- 测试和验证结果只放在最终回复中，不增加报告第四类。

## 验证与固定目标 SHA

- 在不改上游源码的临时 detached worktree 或等价隔离环境中，运行目标 SHA 的上游测试和必要检查。
- 运行利润网现状测试和必要的源码漂移检查；测试的目的在于发现差异，不授权修复差异。
- 若利润网测试仅因源码指纹、快照或旧规则期望而失败，必须定位原因并归入相应报告类别，不得修改利润网来消除失败。
- 这类已定位且由待批准差异造成的利润网失败属于分析结论，不触发恢复旧 SHA。
- 完成增量审查、收益闭环、上游目标 SHA 验证、利润网现状测试、必要的漂移检查和三类报告后，才可把 `game-source/current/` 以 detached HEAD 固定到目标 SHA，并在主仓库记录新的 gitlink、当前固定 SHA 说明和报告索引。
- 若无法完成抓取、增量与闭环分析、上游目标 SHA 测试、利润网现状测试、必要的漂移检查、目标检出或报告，或者利润网检查出现无法归因的失败，必须把 `game-source/current/` 恢复到主仓库原先记录的 commit，不得留下半更新的子模块指针。前述“已定位且由待批准差异造成的利润网失败”除外。

## 后续利润网更新

- 分析报告完成后，默认动作是等待用户判断，不得自动实施报告中的修复。
- 只有用户明确要求修复、同步或更新利润网后，才可修改利润网实现、测试、配置、PRD 或生成产物。
- 后续实现属于独立阶段，必须按用户选择的报告项确定范围，并重新执行与改动风险相称的测试和页面验证。
